### FINDING-001 [MEDIUM] Security invariant S-2 lacks a negative regression test for value-in-error-message leakage

**Location**: `credential-provider/src/vault_tests.rs` — `dynamic_credentials_extractor` module
**Spec Reference**: `docs/spec/security.md §S-2` ("CredentialError messages must never include credential values"), `docs/spec/constraints.md §"Errors are safe"`

**Description**:
The test suite verifies that error messages for missing fields contain the field *name* (e.g., `"username"`, `"password"`), but no test asserts that the field *value* is absent from those messages.

The current implementation of `extract_str_field` is correct — the format string is `"missing field: {field}"` where `{field}` is the field name, never the field value — so no leakage exists today. However, the security invariant that `CredentialError` messages must not include credential values has no sentinel test guarding it. A future refactor that adds the field value to the error message for easier debugging (a common mistake) would pass the existing tests silently.

**Impact**:
Without a negative assertion, a careless change could cause a plaintext password to appear in `CredentialError::Backend` messages, which may be logged by downstream consumers. This violates `S-2` and security invariant #6 ("Errors are safe: CredentialError messages contain paths, status codes, and context — never credential values").

**Reproduction**:
Modify `extract_str_field` to include the value:

```rust
// hypothetical bad change
.ok_or_else(|| CredentialError::Backend(format!("missing field: {field}, got: {}", data[field])))
```

All existing tests would still pass.

**Remediation**:
Add a negative assertion to at least the `missing_password_error_message_contains_field_name` test (and ideally a separate security-specific test), asserting the password value is absent from the error message:

```rust
#[test]
fn missing_password_error_message_does_not_contain_field_value() {
    // The field is present but wrong type — value present, type mismatch
    let sentinel_value = "hunter2-secret-value";
    let data = serde_json::json!({"username": "alice", "password": 42});
    // Use a data object where the value IS present but wrong type (number),
    // so extract_str_field fails but data[field] exists.
    // Assert the sentinel value is not in the error (it isn't a string here,
    // but the key security property is: field values never leak).

    // Simpler variant: test with a *present* but non-string password.
    // The sentinel pattern: use a password value that, if it leaked, would be obvious.
    let data2 = serde_json::json!({"username": "alice", "password": sentinel_value});
    // Force a missing-field error by testing with a field name that doesn't exist.
    // (extract_str_field with field "nonexistent" on a valid object)
    // This confirms the value of other present fields never appears in the error.
}
```

A cleaner approach: directly test `extract_str_field` with a value present but wrong-typed and assert the value does not appear:

```rust
#[test]
fn error_message_never_contains_field_value() {
    // "password" is present in data but as a number (type mismatch → Backend error).
    // The number value must not appear in the error message.
    let data = serde_json::json!({"username": "alice", "password": 99999});
    let result = extractor().extract(&data, None);
    match result {
        Err(CredentialError::Backend(msg)) => {
            assert!(
                !msg.contains("99999"),
                "Field value must never appear in error messages; got: {msg}"
            );
        }
        other => panic!("Expected Err(Backend), got {other:?}"),
    }
}
```

**Spec Compliance**: PARTIAL — implementation is correct; test coverage of the invariant is incomplete.

### FINDING-002 [LOW] Catch-all arm in `map_vaultrs_error` forwards arbitrary vaultrs `Display` output

**Location**: `credential-provider/src/vault.rs` — `map_vaultrs_error`, catch-all arm (`other =>`)
**Spec Reference**: `docs/spec/security.md §S-2`, `docs/spec/constraints.md §"Errors are safe"`

**Description**:
The catch-all arm is:

```rust
other => CredentialError::Backend(format!("vault error: {other}")),
```

This forwards the `Display` output of any unrecognised `vaultrs::ClientError` variant directly into a `CredentialError::Backend` message. The code comment acknowledges this explicitly:

> "If vaultrs 0.9+ adds a variant whose Display leaks internal details, this arm will surface them in CredentialError::Backend messages. Review this arm whenever vaultrs is bumped."

No known vaultrs 0.8.x variant produces a Display output that includes sensitive values (tokens, secrets, credentials). The risk is entirely forward-looking.

**Impact**:
If a future `vaultrs` upgrade introduces an error variant whose `Display` includes a Vault token, path, or credential value, it would flow into the `CredentialError::Backend` message and could be logged by downstream consumers.

**Remediation**:
No code change required for vaultrs 0.8. Ensure the following process controls are in place:

1. Renovate PR for any `vaultrs` bump must include a manual review step confirming no new `ClientError` variants were added whose `Display` output could include sensitive data.
2. Add a comment cross-reference to the renovate config or a `docs/constraints.md` entry as a named tripwire: "Any vaultrs bump must audit the catch-all arm in `map_vaultrs_error`".

**Spec Compliance**: PASS (current implementation only). Risk is forward-looking.

### FINDING-003 [INFO] `cargo deny check` reports four ignore-listed advisories as "not detected"

**Location**: `deny.toml`
**Spec Reference**: `docs/spec/security.md §S-8` (dependency supply chain)

**Description**:
`cargo deny check` emits `advisory-not-detected` warnings for all four advisories in the `deny.toml` ignore block (`RUSTSEC-2026-0097`, `RUSTSEC-2024-0384`, `RUSTSEC-2024-0436`, `RUSTSEC-2025-0134`), while `cargo audit` finds all four as active warnings in the same dependency tree.

This discrepancy is likely due to `cargo deny` and `cargo audit` using different advisory database snapshots or lookup logic. The overall `cargo deny check` result is **pass** (advisories ok, bans ok, licenses ok, sources ok).

**Impact**:
The ignore entries in `deny.toml` are not being exercised by `cargo deny check`. If the advisories are resolved upstream and the ignore entries become permanently stale, it could create false confidence that advisory tracking is active.

**Remediation**:

1. Run `cargo deny check --log-level debug` to confirm whether the discrepancy is a DB-freshness issue or a cargo-deny version issue.
2. If `cargo audit` continues to find these and `cargo deny` does not, prefer `cargo audit` as the authoritative source for advisory checking in CI, and treat `cargo deny` as the licence/ban enforcement tool.
3. No change needed to `deny.toml` until the root cause is confirmed — the advisories may reappear after a `cargo deny` database refresh.

**Spec Compliance**: INFO — no security gap, advisory-tracking hygiene concern only.

## Focused Security Review: `DynamicCredentialsExtractor`

### S-1: Secret material — `SecretString` wrapping

The password field follows this path:

```
JSON Value (&str, borrowed) → .to_owned() → String → SecretString::from(String)
```

The `String` is moved (not copied) into `SecretString`, which wraps it in `Zeroizing<String>`. The `Zeroizing` wrapper zeros the heap allocation on drop. **The password is never a free-standing `String` or `&str` at rest after construction.** ✅

The original `&str` borrow points into the `serde_json::Value` in memory, which is not zeroed. This is the known, documented residual risk from `docs/spec/security.md §S-1` ("Intermediate copies made by the Rust runtime, allocator, or OS page cache are not zeroed"). It predates this task and is an accepted limitation.

### S-2: Logging / debug output paths

- `SecretString` derives its `Debug` impl from `secrecy`, which outputs `"[REDACTED]"`. ✅
- `UsernamePassword::Debug` implementation (in `credential-provider-core`) must not forward the `SecretString` raw value — this was reviewed in previous tasks and verified to use the `secrecy`-provided redaction.
- `extract_str_field` error message is `"missing field: {field}"` — contains field **name** only, never field **value**. ✅
- No `tracing` span, `log!`, or `eprintln!` is present in the new code paths. ✅

### S-3: `client.settings` (Vault token) access

No new code in this task accesses `self.client.settings` or any field that could expose the Vault token. The `VaultClient` is used only as an opaque handle passed to `vaultrs::kv1::get_raw`. ✅

### S-4: Integer safety in `Duration::from_secs(d)`

The `lease_duration_secs: Option<u64>` originates from `lease_secs_from_raw(response.lease_duration)` where `response.lease_duration` is an `i32`. The maximum value is `i32::MAX = 2,147,483,647` seconds (~68 years). `Instant::now() + Duration::from_secs(2_147_483_647)` does not overflow in any practical environment. ✅
